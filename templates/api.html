minbin is a tiny pastebin you can use over plain HTTP.

Pastes are stored in Redis and auto-delete after a fixed expiry. The web UI encrypts pastes in your browser before upload, but the API also accepts plaintext for terminal use (curl, scripts, CI). All endpoints are unauthenticated and rate-limited per IP and (for SRP endpoints) per paste ID — see SECURITY.md for the exact caps.

overview

The base URL is your deployment's domain. There are eight endpoints worth knowing:

methodpathpurpose
POST/create a new paste
GET/:paste_idview paste in the browser (HTML)

quickstart

The fastest way to create a paste – pipe text into curl:

POST https://minb.in/ $ curl --data-binary "hello from the terminal" https://minb.in/ → https://minb.in/a3f9

Pass the URL it prints back, or fetch the body with GET /raw/:id:

GET https://minb.in/raw/a3f9 $ curl https://minb.in/raw/a3f9 → hello from the terminal

POST / – create a paste

Send the paste body as the request body. Any content type is accepted; it's read as raw bytes. The response is the paste URL (or the bare ID if ?relative is set), with status 201.

query parameters

nametypedescription
once flag If present (any value), the paste self-destructs after the first GET /:id or GET /raw/:id. Plaintext pastes only – encrypted pastes carry their own once flag inside the record (see encrypted records).
relative boolean If true, the response is just the paste ID (e.g. a3f9) instead of the full URL.
maxAccesses integer 1..1000 Plaintext only. Auto-deletes after the Nth read of GET /:id or GET /raw/:id. Combine with the time expiry — paste vanishes whichever comes first. Encrypted pastes set this inside the JSON record.

When the deployment has CAPTCHA_BITS > 0, plaintext POSTs require a solved proof-of-work token: X-Captcha-Token: <token> and X-Captcha-Nonce: <nonce>. See GET /captcha for details. Encrypted POSTs are exempt — the SRP+Argon2 cost already raises the per-paste work for an attacker.

examples

# plain string curl --data-binary "secrets are bad" https://minb.in/ # from a file curl --data-binary @notes.txt https://minb.in/ # self-destruct after first read curl --data-binary @notes.txt "https://minb.in/?once" # return the bare ID curl --data-binary "hi" "https://minb.in/?relative=true"
heads-up Plaintext pastes are visible to the server operator. For private content use the web UI – it encrypts in your browser before upload (SECURITY.md).

GET /:paste_id – view in browser

Renders the paste in an HTML page suitable for sharing. For encrypted (v2) pastes this returns the decrypt shell – the actual decryption runs client-side using the key in the URL fragment. For plaintext pastes flagged once, the paste is deleted on this read.

GET https://minb.in/a3f9

GET /raw/:paste_id – fetch raw content

Returns the paste body as text/plain for plaintext pastes, or the encrypted record as application/json for v2 pastes. Designed for scripting.

GET https://minb.in/raw/a3f9 $ curl -s https://minb.in/raw/a3f9 → hello from the terminal

For plaintext once pastes this endpoint also burns on read. Encrypted pastes are not deleted on raw fetch – the client must explicitly call POST /:id/burn after a successful decrypt, otherwise a wrong password would destroy the paste.

POST /:paste_id/burn – delete a paste

Idempotent delete. Returns 204 No Content whether or not the paste existed. Used by the encrypted-paste client to clean up once pastes after decrypt; you can also call it manually to revoke a paste early.

POST https://minb.in/a3f9/burn $ curl -X POST https://minb.in/a3f9/burn -i → HTTP/1.1 204 No Content

POST /:paste_id/access – bump the access counter

Encrypted-paste max-views counter. After a successful client-side decrypt, the client calls this; the server atomically increments the counter and deletes the paste once the cap (the record's maxAccesses field) is reached. Idempotent and safe to retry — a dropped network response can never accidentally re-burn a paste. For plaintext pastes the counter is server-driven on read and this endpoint is a no-op (still returns 200).

POST https://minb.in/a3f9b1c2/access $ curl -X POST https://minb.in/a3f9b1c2/access → {"remaining": 3, "expired": false}

GET /captcha – proof-of-work captcha

Stateless hashcash-style captcha. The server hands you an opaque token + a difficulty in bits. Find a 64-bit nonce such that SHA-256(rand_part_of_token || nonce_be64) has at least bits leading zero bits, then submit on plaintext POST / via X-Captcha-Token + X-Captcha-Nonce headers (or ?captchaToken=&captchaNonce= query). When the deployment has CAPTCHA_BITS=0 (the default in single-user installs) the endpoint still works and returns bits=0; the captcha is bypassed.

GET https://minb.in/captcha $ curl -s https://minb.in/captcha → {"token": "AaWzJ…","bits": 16, "expiresAt": 1714500000000}

No third-party JS, no fingerprinting, no IP leak to a SaaS. Works for curl users with a small bash + python helper. The token has a TTL (CAPTCHA_TTL_SECONDS, default 300s) and is single-use (replay-protected via Redis).

POST /report – abuse report drop-box

Operator-facing abuse reporting. Reports are pushed onto a Redis list with TTL, drained by the operator (typically forwarded to {{ abuse_email }}). Encrypted pastes are opaque to the operator; include the full link (with the URL fragment, the #k=… bit) so they can evaluate content.

POST https://minb.in/report Content-Type: application/json { "paste_id": "a3f9b1c2", "reason": "phishing — fake bank login", "contact": "you@example.com", "link": "https://minb.in/a3f9b1c2#k=…" }

GET /openapi – the spec

Returns the OpenAPI 3 document for this deployment. Useful for generating clients or feeding into Postman/Insomnia.

GET https://minb.in/openapi

encrypted records

To submit an end-to-end encrypted paste from your own client, send a JSON record with Content-Type: application/vnd.minbin.encrypted+json. The server validates the record's shape and stores it verbatim; it cannot decrypt it. The full wire format, key derivation parameters, and threat model are documented in SECURITY.md.

In short: AES-256-GCM payload with a 12-byte IV, optionally wrapped by an Argon2id-derived key. The once flag lives inside the record (so a curious server can't lie about it), and the random 256-bit master key never leaves the browser – it travels in the URL fragment #k=…, which browsers don't send to servers.

errors & limits

statuswhen
400Empty body, body over the size limit, or a malformed encrypted record.
402Daily per-IP byte budget exceeded (only when DAILY_BYTE_BUDGET > 0). Honor the Retry-After header — it counts seconds to UTC midnight.
404The paste ID doesn't exist or has expired.
403Cross-origin request blocked: an Origin or Sec-Fetch-Site header inconsistent with APP_DOMAIN was present on a write.
422Query parameter validation failed, or the captcha was missing/rejected.
429Rate-limited. Per-IP and per-paste-ID buckets; honor the Retry-After header.
500Unexpected server error (and the only time you should retry blindly).

Two configurable limits matter: MAX_PASTE_SIZE (default {{ max_paste_size_mb }} MB) caps the request body, and PASTE_EXPIRY (default {{ paste_expiry_minutes }} minutes) is how long a paste lives in Redis. Both are set per deployment.
